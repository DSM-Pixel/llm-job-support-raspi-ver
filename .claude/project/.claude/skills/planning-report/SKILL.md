# /planning-report

기획 문서 초안을 만든다. 빈 종이에서 시작하지 말고 양식을 채워준다. 저장 위치는 항상 `docs/`.

## 입력 수집

`AskUserQuestion`으로:
1. **문서 종류**: 기획 보고서 / 기능 정의서 / 아키텍처 설계안 / 발표자료
2. **대상 기능/파트** (모르면 `backend/`·`prototypes/`·`docs/`를 스캔해 선택지로 보여준다 — 예: 공공데이터 RAG, 이미지 라벨링, 통합 플랫폼 전체)
3. **시연 시나리오 한 줄** (예: "포트홀 영역을 빨갛게 표시해줘")

## 생성 위치

```
docs/
├── <파트>-<문서종류>.md           # 예: 공공데이터RAG-기획보고서.md
└── slides/<제목>.md               # 발표자료는 slides/ 아래 (Marp 호환)
```

이미 같은 경로가 있으면 **덮어쓰지 말고 `-v2.md` suffix**를 붙인다.
비슷한 문서가 `docs/`에 이미 있으면 (예: `기능명세서_이미지라벨링.html`) 새로 만들지 말고 그 내용과 정합성을 맞춘다.

### 2) 기능 정의서

표 형식 위주. 한 기능당 행 하나. 각 기능에 **시연 시나리오 1줄** 필수.

### 3) 아키텍처 설계안

- 컴포넌트 다이어그램(ASCII) + 컴포넌트별 한 단락 설명
- 데이터 흐름(요청 → 응답까지)
- 외부 의존성(Gemini API, 공공데이터 API, SQLite 스토리지)

## 작성 가이드

- 모든 섹션을 비워두지 말 것. 모르면 `<TODO: …>` 마커로라도 표시.
- 표는 가능하면 5행 이하 (가독성). 발표자료 슬라이드당 **5줄 이내**.
- 오픈소스 대회 출품 문서에는 라이선스를 **AGPL-3.0**으로 명기 (변경 금지).
- 실제로 확인 안 한 데이터셋/모델 이름을 쓰지 말 것 (`public-data-finder`·`vlm-researcher`로 확인).

## 마지막 행동

1. 생성한 파일 경로 알림.
2. `planning-writer` 에이전트를 부르면 본문을 채워주겠다고 안내.
3. 빠진 정보(데이터셋 이름, 성능 수치 등)는 체크리스트로 정리.
